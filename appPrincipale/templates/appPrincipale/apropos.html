Histoire

Ce site a été créé dans le cadre d'un projet de développement WEB.

Mission

AdopteUnTruc est un site permettant l'échange d’objet ou bien même le proposition de services. Le principe étant que des personnes prêtent à d’autre utilisateurs des objets qu’ils n’utilisent pas souvent.

Par exemple sur un campus universitaire pourvu de nombreux terrains de tennis nous avons pensé qu’il serait intéressant de pouvoir être mis en relation avec des personnes possédant des raquettes de tennis afin de pouvoir faire une petite partie sans avoir besoin d’acheter tout le matériel nécessaire.

Fonctionnement général

Pour atteindre nos objectifs, nous avons défini plusieurs objectifs principaux, vitaux au bon fonctionnement, et des objectifs secondaires, qui seraient optionnels mais qui pourraient être de véritables atouts pour notre site.

Fonctionnalités principales

  • Gestion de comptes utilisateurs. Un utilisateur doit pouvoir s’inscrire, se connecter, se déconnecter.
    L’utilisateur a aussi la possibilité de modifier ses informations personnels tels que nom, prénom, email. Cependant il n’est pas possible de changer son mot de passe, ni de le récupérer en cas de perte.
  • Article : lorsqu’un utilisateur est connecté il peut créer un objet, le modifier, ou le supprimer.
    Ainsi tous les objets créée sont forcément lié à un compte utilisateur.
  • Recherche d’objets : il est possible de faire des recherches sur plusieurs critères :
    • recherche de contenu dans le titre et la description de l’article
    • recherche par code postal
    • recherche par Catégorie

Fonctionnalités secondaires

  • Mise en place d’un système commentaires. Chaque utilisateur connecté avec son compte à la possibilité d’ajouter un commentaire à un article.
    Ainsi les commentaires sont liés à un article et aussi à un utilisateur.
  • Cartographie : Lorsque l’on effectue une recherche tous les articles correspondant à la requête seront affichés sur une carte grâce à l’API Google. De plus lorsqu’un utilisateur se connecte il a la possibilité d'accepter d’être géolocalisé. La carte sera automatiquement centrée sur la position de l’utilisateur. Mais si ce dernier refuse, la carte sera centrée sur la France.
  • Filtre de données : nous avons mis en place des filtres empêchant un utilisateur de saisir des propos injurieux dans les commentaires et les articles. De plus l’utilisateur est aussi obligé de saisir un code postal valide.
  • Mise en place d’un script Javascript permettant de mettre directement l'utilisateur sur la portion de page contenu le résultat de sa recherche.

Technologies utilisées pour le développement

Nous avons utilisé le framework Django pour la création de notre site. Ce framework comporte de nombreux intérêts puisqu’il fournit de nombreuses fonctionnalitées préconçues et notamment l'intégration d’un ORM permettant la communication avec n’importe qu’elle base de donné. Django va encore plus loin puisqu’il intègre directement une base de données Sqlite3.

Mais le véritable point fort de cet framework c’est qu’il utilise l'architecture logicielle MVT (Model View Template), dérivé de MVC (Model View Controller), ce qui facilite grandement le développement, la compréhension de l’architecture. C’est aujourd’hui quasiment indispensable d’utiliser cette architecture.

Concernant la partie la visuelle HTML/CSS, nous avons utilisé un bootStrap gratuit, que nous avons adapté au fonctionnement de la framework Django. Ce bootStrap, en plus d’avoir un visuel agréable avait aussi la caractéristique d’être Responsive.

Pour l’intégration de la carte et ses différentes fonctionnalitées (géolocalisation de l’utilisateur, des articles…), nous avons utilisé l’API GoogleMaps en JavaScript.

Base de données

Django passant par un ORM, la gestion de la base de donnée n’est pas un problème. Nous n’avons pas eu à écrire de requêtes SQL, et l’ajout, suppression de données se gère comme avec des objets. Nous n’avions qu'à définir des modèles pour ces objets, que nous définissons ci-dessous.

Comme on peut le voir de part les dépendances fonctionnelles (représentées par des flèches bleues) ces “modèles” correspondent à des tables FNBC.

Le seul doute que nous avons pu avoir concernait la dépendance fonctionnelle entre le code postal et l’adresse, cependant vu que plusieurs ville on les mêmes noms de rue, cette dépendance n’existe pas.

L’une des particularités des modèles de Django est que les clefs étrangères correspondent directement à des objets des autres modèles, c’est pour cela que nous avons mis Objet, User et non user-id et id-objet.

Une particularité est que les “id” sont des clés uniques, générées automatiquement par le framework et donc ils n’apparaissent pas dans la définition de nos modèles.

Architecture MVT

Models

Le framework Django a développé ses propres fonctions de recherche, ces dernières sont beaucoup plus facile à prendre en main que des requêtes SQL classique mais elles nécessitent cependant que les “modèles d’objets” soient fait selon les règles de construction de Django.

Par exemple si l’on veut sélectionner tous les articles publiés par le User=1, il n’est pas nécessaire de faire une jointure de table Django s’en charge. Ainsi la requête ressemblera à : Comentaire.objects.filter(user_id=1)

Validation des données

Pour insérer des informations dans la base de données, Django met a disposition des fonctions vérifiant les informations passées dans les formulaires (forms.py). Nativement, cela permet seulement de vérifier que le format des données en entrées est identique au format des données attendus par le modèle. Par exemple un champ “Prénom” rempli par “12435” sera refusé.

Nous avons cependant ajouté nos propres critères de validation. Par exemple il n’est pas possible de publier de gros mots, ni de rentrer un code postal qui ne soit pas composé de cinq chiffres.

Vue

La Vue est le seul élément visible par l’utilisateur. C’est l’interface graphique entre l’utilisateur et le controller.

Template/"Controller"

Le controller est censé prendre en charge tout les interactions de l’utilisateurs, d'interagir avec le base de donnée, puis avec la vue qui renverra les informations à l’utilisateur.

Cependant Django utilise le modèle MVT, remplaçant le C de controller par le T de template. En effet le développeur n’a pas besoin de s’occuper du controller, tout est orchestré par Django et nous n’avons qu'à fournir des templates HTML avec des champs prédéfini que Django remplira lui même en fonction de nos modèles.

Maquette du site

L’entête de notre site est un menu bandeau permettant de naviguer entre les différentes pages de notre site. Dans le pied de page ce trouve un autre menu permettant cet fois ci d’avoir accès au caractéristique de notre site notamment : “le rapport de projet” et “les termes d’utilisation”

AdopteUnTruc

Cette page est la page d’accueil de notre site. On peut y trouver une barre de recherche ainsi que tous les articles ayant été publiées pendant le mois courant.

Tous les articles

C’est sur cette page que sont affichés les résultats d’une recherche (par défaut ce sont les objets publiés pendant le mois qui sont affichés). Tous les objets de la recherche sont affichés sur une carte google map avec leurs distance par rapport à l’utilisateur.

Profil

Sur cette page s’affiche tous les objets que l’utilisateur a partagé sur AdopteUnTruc. S’il clic sur un objet il sera redirigé sur une page où il pourra supprimer ou modifier sa publication s’il le souhait. De plus depuis “Profil” l’utilisateur peut modifier son profil s’il le souhaite. Si l’utilisateur n’est pas connecté avec son compte il est rédigé vers une page de connection.

Inscription

Depuis cette page l’utilisateur peut se créer un compte.Des conditions sur le mots de passe et l’adresse mail ont été mis en place. Ainsi un mot de passe trop faible en terme de sécurité sera refusé, ainsi qu’une adresse mail erronée.

Connexion

Sur cette page l’utilisateur peut se connecter avec un compte déjà créé.

Ajouter un article

ci l’utilisateur ajoute un article. L’article sera automatique lié au compte de l’utilisateur, de plus une date de publication est aussi ajouté automatiquement. Si l’utilisateur n’est pas connecté avec son compte il est rédigé vers une page de connection.

A améliorer

Nous n’avons pas mis en place de système permettant à l’utilisateur de changer son mot de passe, ni de le récupérer en cas d’oublie. De plus nous n’avons fait que très peu d'Ajax ce qui est dommage étant donné que cette méthodologie est très utile, notamment pour l’envoie de formulaire par exemple. Il aurait aussi été très intéressant que les autres utilisateurs ne puissent pas avoir accès à l’adresse mail du propriétaire d’un objet et que tous les échanges d’informations se fasse via la site. Néanmoins étant donné que Leboncoin fonctionne de cette manière nous n’avons pas pensé qu’il était fondamentale d’implémenter cette fonctionnalité.
